# PRD: Fix Sidebar Category Ordering and Uncategorized Page Placement

## Status

**Date:** 2026-06-11
**Author:** PRD
**Audience:** Engineering team

---

## 1. Problem Statement

After `docs:pull` generates correct `_category_.json` with proper Notion `Order`-based positioning, the browser still shows wrong sidebar ordering and uncategorized pages appear as top-level sidebar links.

Two distinct issues affect the sidebar:

### Issue A: Stale `_category_.json` in comapeo-docs

Our pipeline generates correct `_category_.json` with proper `position` values at the `docs:pull` output directory. But comapeo-docs has its own `docs/*/_category_.json` files from the OLD pipeline that are never overwritten. When a developer copies the new files over, the ordering works. There is no automated integration step.

**Observed ordering (old, wrong):**
```
pos=1 Managing Data & Privacy         (order: 4)
pos=2 Gathering Observations & Tracks (order: 3)
pos=3 Overview                        (order: no toggle)
pos=4 Miscellaneous                   (order: 23)
```

**Correct ordering (generated by pipeline):**
```
pos=1 Gathering Observations & Tracks  (order: 3)
pos=2 Managing Data & Privacy          (order: 4)
pos=3 Reviewing Observations & Tracks  (order: 11)
pos=4 Exchanging Observations          (order: 11)
pos=5 Miscellaneous                    (order: 23)
pos=6 Preparing to use CoMapeo         (order: 32)
pos=7 Managing Projects                (order: 32)
pos=8 Sharing and Exporting...         (order: 999, no toggle)
pos=9 Overview                         (order: 999, no toggle)
```

### Issue B: Uncategorized pages at top level

Pages with `section: null` (no `Content Section` property set in Notion) are written directly to `docs/{slug}.md` instead of `docs/{section}/{slug}.md`. Docusaurus places these files as top-level sidebar items outside any category.

**Example:** "Using Exchange Over the Internet" appears in all 3 locales as a top-level link in the sidebar. It is a real content page with no `Content Section` property in Notion.

### Issue C: Section labels truncated in translated locales

Toggle page titles for PT locale are truncated in Notion (e.g., "Tracar Observao..." instead of full "Trocar Observacoes e Trayectos"). The static `SECTION_TRANSLATIONS` map in `src/cli/index.ts` provides correct full labels but some may be missing or outdated.

### Issue D: No integration script between pipeline output and comapeo-docs

There is no documented, automated step to run `docs:pull --all --clean-orphans` and copy the output into comapeo-docs. Developers manually copy files, which is error-prone.

---

## 2. Root Cause Analysis

### 2.1 Stale `_category_.json` — Root Cause

The pipeline generates `_category_.json` at `cmdDocsPull` in `src/cli/index.ts` (lines 514-550). Each category file includes:

```json
{
  "label": "Gathering Observations & Tracks",
  "position": 1,
  "collapsible": true,
  "collapsed": true,
  "link": { "type": "generated-index", "title": "Gathering Observations & Tracks" },
  "customProps": { "title": "Gathering Observations & Tracks" }
}
```

The `position` field is assigned by sorting sections via `sectionOrder` map (built from Notion's EN Toggle `Order` property), then incrementing from 1. This sorting is correct.

However, comapeo-docs has its own copy of these files at `docs/{section}/_category_.json` from a previous manual setup. Running `docs:pull` writes to a separate output directory. No step copies those files into the comapeo-docs repository.

**Affected files in comapeo-docs:**
- `docs/*/_category_.json` — each section directory has one
- `i18n/es/docusaurus-plugin-content-docs/current/*/_category_.json`
- `i18n/pt/docusaurus-plugin-content-docs/current/*/_category_.json`

### 2.2 Uncategorized Pages — Root Cause

In `cmdDocsPull` (line 474), pages with `section: null` get `sectionRaw = undefined`, which causes `sectionDir = undefined`. This writes the page to:

```
docs/{slug}.md            (EN)
i18n/{locale}/.../{slug}.md  (non-EN)
```

Instead of to a section subdirectory. Docusaurus autogenerated sidebars pick up orphan `.md` files at the root level.

**Affected page:**
- "Using Exchange Over the Internet" — has `section: null` in all locales
- 72 untranslated EN pages also have no Content Section in Notion

### 2.3 Static Translation Map Gaps

The `SECTION_TRANSLATIONS` map (lines 876-906) has hardcoded label pairs. It was manually assembled and may not match the current section names extracted from Notion. If a section name changes in Notion, the translation falls through to the stripped English name.

### 2.4 No Integration Script

The `docs:pull` command accepts `--out` to specify the output directory but the canonical workflow (point it at comapeo-docs) is not codified as a script or recipe.

## 3. Constraints

- Must not modify Notion data (cannot add `Content Section` properties to uncategorized pages)
- Must work with Docusaurus 3.9.2 sidebar autogeneration (category items and `sidebar_position` in frontmatter)
- Must work for all 3 locales (EN, ES, PT)
- `docs:pull --all --clean-orphans` must be the canonical update command
- The Docusaurus `sidebarItemsGenerator` fix (from PRD-sidebar-i18n.md) handles sidebar label translations — this PRD focuses only on ordering and page placement
- Must not break existing pages or create duplicate sidebar entries
- Integration step must be automated (scriptable, not manual copy)

## 4. Proposed Solutions

### 4.1 Solution A: Integration Script + Pipeline Fix

#### Step 1 — Add category override script for comapeo-docs

Create a shell script `scripts/sync-docs.sh` (or similar) that:

1. Runs `bun src/cli/index.ts docs:pull --all --clean-orphans --out /tmp/comapeo-docs-output`
2. Checks out the latest `main` on comapeo-docs
3. Overwrites `docs/` and `i18n/` directories with the generated output
4. Shows a diff of changed `_category_.json` files for review
5. Guides the developer to commit

**Minimal Bash script:**

```bash
#!/usr/bin/env bash
set -euo pipefail

OUTDIR=$(mktemp -d)
COMAPEO_DOCS="${COMAPEO_DOCS_PATH:-../comapeo-docs}"

bun src/cli/index.ts docs:pull \
  --input output/manifest.json \
  --input-dir output \
  --out "$OUTDIR" \
  --all \
  --clean-orphans

# Overwrite comapeo-docs content
for locale_dir in "$OUTDIR"/docs "$OUTDIR"/i18n; do
  target="$COMAPEO_DOCS/$(basename "$locale_dir")"
  rsync -a --delete "$locale_dir/" "$target/"
done

echo "Sync complete. Review changes in $COMAPEO_DOCS"
cd "$COMAPEO_DOCS" && git diff --stat
```

Alternatively, make this a CLI subcommand in `src/cli/index.ts` to avoid shell scripting.

#### Step 2 — Fix uncategorized pages (assign to fallback section)

In `cmdDocsPull` (around line 474), change the section resolution for Docusaurus output paths so that pages with null section get assigned to a fallback section (e.g., "Overview" or "Uncategorized"). This ensures they appear inside a sidebar category, not at the top level.

**Current code (line 474):**
```ts
const sectionRaw = effectiveSection || undefined;
const sectionDir = sectionRaw ? toSectionDir(sectionRaw) : undefined;
```

**Proposed change:**
```ts
const sectionRaw = effectiveSection || "__uncategorized__";
const sectionDir = sectionRaw ? toSectionDir(sectionRaw) : undefined;
```

This alone is not enough — we also need to generate a `_category_.json` for the fallback section and decide where it appears in the sidebar order.

**Better approach:** Assign `__uncategorized__` pages to the "Overview" section (if it exists) or create a new section at the highest position (large order number, e.g., 999). This is a decision that requires looking at the available sections:

- "Overview" exists in the pipeline with no Toggle (position 999, no explicit order). This is a natural fallback.
- Alternatively, create an "Uncategorized" section at the end of the sidebar.

**File paths affected:**
- `src/cli/index.ts` — `cmdDocsPull` section resolution (line ~474)
- `src/cli/index.ts` — `toSectionDir` function (line ~914) — needs to handle `__uncategorized__`
- `src/cli/index.ts` — `assignFallbackPositions` (line ~937) — needs to handle null-section metadata
- Translation map — may need entries for the chosen fallback label

#### Step 3 — Add missing section labels to SECTION_TRANSLATIONS

Audit the current sections extracted from Notion and update the static map:

```ts
// Check which sections exist by running:
// grep 'section:' from manifest or inspect sectionPositions map

// Detect sections without translation entries
function auditSectionTranslations(manifest: Manifest) {
  const sectionNames = new Set<string>();
  for (const doc of manifest.docs) {
    if (doc.section) sectionNames.add(doc.section);
  }
  for (const name of sectionNames) {
    const stripped = stripSectionPrefix(name);
    for (const locale of ['pt', 'es']) {
      if (!SECTION_TRANSLATIONS[locale]?.[stripped]) {
        console.warn(`Missing ${locale} translation for section: "${stripped}"`);
      }
    }
  }
}
```

### 4.2 Solution B: Enable `docs:pull --out` to target comapeo-docs directly

Modify `cmdDocsPull` to accept `--repo` pointing to comapeo-docs checkout, then clean and overwrite in-place:

```
bun src/cli/index.ts docs:pull --repo ../comapeo-docs --all --clean-orphans
```

This replaces the manual copy step with a single command. Under the hood it:
1. Generates docs to a temp directory as before
2. Removes `docs/` and `i18n/` from the target repo
3. Copies the new files in
4. Leaves `git status` for the developer to review

### 4.3 Recommended Approach: Solution A (simple script) + pipeline fixes

Solution B adds complexity to the CLI that belongs in a shell wrapper. Keep the pipeline focused on generation; use a script for deployment.

**Priority:**
1. P0: Fix uncategorized page placement (Section 4.1 Step 2)
2. P0: Create integration script (Section 4.1 Step 1)
3. P1: Audit and fill SECTION_TRANSLATIONS (Section 4.1 Step 3)
4. P2: Add automated test for sidebar structure correctness

## 5. Implementation Plan

### Phase 1 — Fix uncategorized pages

**File:** `src/cli/index.ts` — `cmdDocsPull`

**Change 1a:** Add fallback section constant and helper function.

```ts
const FALLBACK_SECTION = "Uncategorized";
const FALLBACK_SECTION_ORDER = 999;

function resolveSection(section: string | null, effectiveSection: string | null): string | undefined {
  // If we have an effective section (translation pointing to English section), use it
  if (effectiveSection && effectiveSection !== "__uncategorized__") return effectiveSection;
  // If the page has a native section, use it
  if (section) return section;
  // Fallback: assign to "Uncategorized" section
  return FALLBACK_SECTION;
}
```

**Change 1b:** Apply resolution where section is used for file paths (line ~474).

```ts
const sectionRaw = resolveSection(doc.section, effectiveSection);
const sectionDir = toSectionDir(sectionRaw);
```

**Change 1c:** Ensure `__uncategorized__` key name used internally doesn't leak into filesystem paths (use a display name for `_category_.json` label).

**Change 1d:** Add `_category_.json` generation for the fallback section. The `sectionPositions` map will only have entries for pages that had explicit positions. We need to ensure the fallback section also gets a `_category_.json`:

- If any page resolves to the fallback section, add it to `sectionPositions` with a high order number (999).
- Get its translated label from `SECTION_TRANSLATIONS` or use the English fallback.

**Change 1e:** Update `assignFallbackPositions` to handle null-section pages that now have a section. The function currently groups by `m.section ?? ""` — with the resolution above, null-section pages will have `"Uncategorized"` as their section by the time `assignFallbackPositions` runs.

**Test plan:**
1. Run `docs:pull` against current manifest
2. Verify "Using Exchange Over the Internet" appears inside "Uncategorized" section
3. Verify uncategorized section appears at the end of the sidebar
4. Check all 3 locales

### Phase 2 — Integration script

**File:** `scripts/sync-to-comapeo-docs.sh`

```bash
#!/usr/bin/env bash
# Sync pipeline output to comapeo-docs repository
#
# Usage:
#   COMAPEO_DOCS_PATH=../comapeo-docs ./scripts/sync-to-comapeo-docs.sh
#   COMAPEO_DOCS_PATH=../comapeo-docs ./scripts/sync-to-comapeo-docs.sh --diff-only

set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "$0")/.." && pwd)"
COMAPEO_DOCS="${COMAPEO_DOCS_PATH:-../comapeo-docs}"
OUTDIR=$(mktemp -d)
trap 'rm -rf "$OUTDIR"' EXIT

echo "Pulling docs from pipeline..."
cd "$SCRIPT_DIR"
bun src/cli/index.ts docs:pull \
  --input output/manifest.json \
  --input-dir output \
  --out "$OUTDIR" \
  --all \
  --clean-orphans

echo "Syncing docs to $COMAPEO_DOCS..."
for dir in "docs" "i18n"; do
  if [ -d "$OUTDIR/$dir" ]; then
    rsync -a --delete "$OUTDIR/$dir/" "$COMAPEO_DOCS/$dir/"
  fi
done

echo "Done. Changed files:"
cd "$COMAPEO_DOCS" && git diff --stat -- docs/ i18n/

if [ "${1:-}" = "--diff-only" ]; then
  git diff -- docs/ i18n/
fi
```
