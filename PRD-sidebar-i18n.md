# PRD: Fix Docusaurus Sidebar Category i18n

## Status

**Date:** 2026-06-11
**Author:** PRD
**Audience:** Engineering team

---

## 1. Problem Statement

Docusaurus 3.9.2 generates sidebar category labels from `_category_.json` files. When a locale-specific `_category_.json` exists with a translated `label` (e.g., `"Visao Geral"` for PT), Docusaurus ignores it and uses the default locale's label (e.g., `"Overview"`).

This affects two surfaces:
1. **Sidebar category labels** -- all locales show English
2. **Generated-index category pages** (the `<h1>` title on category index pages) -- all locales show English

### Current output

Our pipeline generates correct translated `_category_.json` at:
- `docs/{section}/_category_.json` (EN -- works)
- `i18n/pt/docusaurus-plugin-content-docs/current/{section}/_category_.json` (PT -- ignored)
- `i18n/es/docusaurus-plugin-content-docs/current/{section}/_category_.json` (ES -- ignored)

Each file has the correct translated `label`, `link.title`, and `customProps.title`.

---

## 2. Root Cause

`@docusaurus/plugin-content-docs` loads `_category_.json` files via `readCategoriesMetadata()` in `lib/sidebars/index.js` (line 90):

```js
const categoriesMetadata = await readCategoriesMetadata(options.version.contentPath);
```

This only scans `${version.contentPath}/**/_category_.{json,yml,yaml}` -- which resolves to `<siteDir>/docs/**/_category_.{...}` for all locales. It never checks `version.contentPathLocalized` (the i18n directory).

By contrast, doc `.md` files **are** correctly localized because `readDocFile()` uses `getContentPathList(versionMetadata)` which returns `[contentPathLocalized, contentPath]` -- the localized path takes priority.

**Summary:** Docusaurus reads translated doc files but not translated `_category_.json` for the same locale.

### Translation file approach (sidebar translations) also doesn't work for autogenerated sidebars

Docusaurus has a translation file system (`i18n/*/plugin-docs.json`) that can translate sidebar labels. However, this system translates sidebar items based on their **`key` attribute** (or `label` as fallback). For autogenerated sidebars (`type: autogenerated`), category keys are auto-derived from the English `_category_.json` label. When `translateContent` runs, it finds the sidebar category by its English key and applies the translation.

In practice, this means:
- The translation file approach **would work** for sidebar labels IF a translation key is properly set up
- But `link.title` on generated-index pages is NOT covered by the sidebar translation system (see `translations.js` lines 60-68 -- only `link.title` and `link.description` for `generated-index` type are included in the translation keys)
- The translation file approach requires running `docusaurus write-translations` and manually maintaining translation JSON files, which is fragile

---

## 3. Constraints

- Must work with Docusaurus 3.9.2 (can't fork Docusaurus core for a shared team project)
- Must work for both `docusaurus start` (dev server) and `docusaurus build` (production)
- Must be maintainable -- prefer approaches that don't break on Docusaurus minor upgrades
- The fix should be in our content pipeline (generation side) OR as a minimal Docusaurus plugin/config change
- Must not require manual translation file maintenance

## 4. Solution Options

### Option A: Docusaurus plugin that patches `readCategoriesMetadata` -- RECOMMENDED

**Approach:** Create a small Docusaurus plugin that monkey-patches the `readCategoriesMetadata` call in `@docusaurus/plugin-content-docs` to also scan `contentPathLocalized` and merge results, with localized entries taking priority.

**How it works:**
1. A plugin with `name: 'fix-sidebar-i18n'` intercepts plugin lifecycle via `extendCli` or overrides the content loading
2. After the docs plugin loads and processes sidebars, and after `translateContent` runs, the fix applies locale-specific `_category_.json` labels for the active locale
3. A simpler variant: use a `postBuild` or `contentLoaded` hook to rewrite the category-generated-index metadata and sidebar data

**Detailed mechanism:**

The cleanest insertion point is a plugin that hooks into the content loading pipeline. Since Docusaurus loads content and then translates it, the approach is:

1. In `contentLoaded`, after the docs plugin has resolved sidebars, read the locale-specific `_category_.json` files
2. Walk the sidebar tree in the loaded content
3. For each category, check if `{contentPathLocalized}/{sectionDir}/_category_.json` exists
4. If so, override the category label, link.title, and customProps.title

```ts
// docusaurus-plugin-fix-category-i18n/index.ts
import type { Plugin } from '@docusaurus/types';
import path from 'path';
import fs from 'fs-extra';

export default function pluginFixCategoryI18n(): Plugin {
  return {
    name: 'fix-category-i18n',
    async contentLoaded({ content, actions, contentRaw }) {
      // No-op: the actual fix needs to happen after sidebar resolution
      // We use `postBuild` and `configureWebpack` as fallback, 
      // or better: use the translateContent hook
    },
  };
}
```

**A more concrete approach** uses the `translateContent` lifecycle which Docusaurus calls per-locale. Since `translateLoadedContent` in `translations.js` already walks the sidebar tree, we can add our own step after it.

However, this is still fragile because we'd be reaching into internals.

**Better approach -- Swizzle or shadow the sidebar generator:**

Docusaurus 3.x allows a custom `sidebarItemsGenerator` via config. The config in `comapeo-docs/docusaurus.config.ts` passes `sidebarPath: './src/components/sidebars.ts'` to the preset. We can pass a custom `sidebarItemsGenerator` function that:
1. Calls the default generator (which correctly reads docs and English `_category_.json`)
2. After generation, walks the resulting sidebar items
3. For each category, reads the locale-specific `_category_.json` from `{contentPathLocalized}/{sectionDir}/_category_.json`
4. Overrides `label`, `link.title`, `customProps.title` with the localized values
5. Returns the modified sidebar items

```ts
// In docusaurus.config.ts, inside the docs preset config:
sidebarItemsGenerator: async ({item, ...params}) => {
  const defaultGenerator = params.defaultSidebarItemsGenerator;
  const sidebarItems = await defaultGenerator({item, ...params});
  return applyLocalizedCategoryLabels(sidebarItems, params);
}
```

The `applyLocalizedCategoryLabels` function:
1. Receives `params.version` which has `contentPath` and `contentPathLocalized`
2. Walks the sidebar items tree
3. For each category, constructs the relative path from the `source` field
4. Reads `{contentPathLocalized}/{source}/_category_.json`
5. If it exists, merges its `label`, `link.title`, `customProps.title` into the category item

```ts
async function applyLocalizedCategoryLabels(items, params) {
  const { version } = params;
  if (!version.contentPathLocalized) return items; // EN locale, nothing to do
  
  return Promise.all(items.map(async (item) => {
    if (item.type === 'category' && item.source) {
      const localizedCategoryPath = path.join(
        version.contentPathLocalized,
        item.source,
        '_category_.json'
      );
      if (await fs.pathExists(localizedCategoryPath)) {
        const localizedMeta = await fs.readJSON(localizedCategoryPath);
        if (localizedMeta.label) item.label = localizedMeta.label;
        if (localizedMeta.link?.title && item.link) {
          item.link = { ...item.link, title: localizedMeta.link.title };
        }
        if (localizedMeta.customProps?.title) {
          item.customProps = { ...item.customProps, title: localizedMeta.customProps.title };
        }
      }
      // Recursively process children
      if (item.items) {
        item.items = await applyLocalizedCategoryLabels(item.items, params);
      }
    }
    return item;
  }));
}
```

**Pros:**
- Minimal changes to `docusaurus.config.ts` (add `sidebarItemsGenerator`)
- No filesystem changes in our pipeline
- Works for both `start` and `build`
- Uses Docusaurus's documented extension point (`sidebarItemsGenerator`)
- Survives `@docusaurus/plugin-content-docs` upgrades (the API is stable)
- No forking or monkey-patching Docusaurus internals

**Cons:**
- Async reads of filesystem during sidebar generation (negligible perf impact for our ~50 pages)
- Must handle the case where `contentPathLocalized` doesn't exist (fall through)

**Effort: 1-2 hours**

### Option B: Pipeline-level sidebar computation (static sidebars)

**Approach:** Instead of using `autogenerated` sidebars, have our `docs:pull` command generate a complete `sidebars.ts` file with all translated labels already baked in. This bypasses `_category_.json` entirely.

**How it works:**
1. `docs:pull` reads the manifest and generates a static `src/components/sidebars.ts` file (instead of just relying on `_category_.json`)
2. The sidebar structure is explicitly defined with all locale-specific labels
3. Docusaurus reads the static sidebar and has no need to look at `_category_.json` for labels

**Pros:**
- Complete control over sidebar content
- No Docusaurus plugin changes needed
- Labels are resolved at pipeline time, not build time

**Cons:**
- Invasive change to `docs:pull` and Docusaurus config
- Sidebar positions become static -- adding/removing pages requires re-running `docs:pull`
- Loses Docusaurus's auto-sidebar hygiene (warns on missing docs, detects orphans)
- More complex to maintain than a small plugin hook

**Effort: 4-6 hours**

---

### Option C: Patch `readCategoriesMetadata` via `patch-package`

**Approach:** Use `patch-package` to apply a small patch to `@docusaurus/plugin-content-docs` that fixes `readCategoriesMetadata` to also scan `contentPathLocalized`.

**Patch diff:**
```diff
// In readCategoriesMetadata function, line 40
async function readCategoriesMetadata(contentPath) {
+  const contentPaths = [contentPath, ...(version.contentPathLocalized ? [version.contentPathLocalized] : [])];
+  const allCategories = {};
+  for (const cp of contentPaths) {
+    const files = await glob('**/_category_.{json,yml,yaml}', { cwd: cp });
+    // ... merge with localized taking priority
+  }
+  return allCategories;
}
```

Actually, the simpler patch is in `loadSidebars` (line 90):
```diff
- const categoriesMetadata = await readCategoriesMetadata(options.version.contentPath);
+ const contentPaths = [options.version.contentPath];
+ if (options.version.contentPathLocalized) {
+   contentPaths.unshift(options.version.contentPathLocalized);
+ }
+ let categoriesMetadata = {};
+ for (const cp of contentPaths) {
+   const cpMeta = await readCategoriesMetadata(cp);
+   categoriesMetadata = { ...categoriesMetadata, ...cpMeta };
+ }
```

**Pros:**
- Fixes the root cause in Docusaurus core
- All other locale-specific metadata from `_category_.json` (collapsible, collapsed, etc.) also work
- No changes needed to our pipeline or Docusaurus config

**Cons:**
- `patch-package` patches break on every `@docusaurus` upgrade
- Need to verify the patch applies cleanly for all team members
- Creates a maintenance burden for upgrades
- Not a supported Docusaurus extension mechanism

**Effort: 30 minutes (patch) + ongoing maintenance**

---

### Option D: Sidebar translation entries (docusaurus write-translations)

**Approach:** Generate per-locale `i18n/{locale}/plugin-docusaurus-plugin-content-docs/current.json` that contains sidebar translation entries for all category labels.

**How it works:**
1. Run `docusaurus write-translations --locale pt` to generate translation JSON files
2. Our pipeline would inject the translated category labels into these JSON files
3. Docusaurus's `translateContent` applies the translations to sidebar items

**Pros:**
- Uses Docusaurus's official i18n path
- No code changes beyond pipeline generation of translation files

**Cons:**
- Docusaurus sidebar translations in autogenerated mode are keyed by the English label, which can collide
- Requires generating sidebars-first then translations -- a chicken-and-egg problem
- `link.title` for generated-index pages is NOT covered by sidebar translation entries (see `translations.js` lines 60-68)
- Still won't fix category index page `<h1>` titles
- Translation JSON files become stale easily

**Effort: 2-3 hours (likely incomplete)**

---

## 5. Recommendation

**Option A (custom `sidebarItemsGenerator`) is the recommended approach.**

Rationale:
- Uses Docusaurus's documented `sidebarItemsGenerator` API (stable since Docusaurus 2.x)
- Does not require patching Docusaurus core
- Does not require changes to our content pipeline (`docs:pull` continues generating `_category_.json` as it does today)
- Fixes both sidebar labels AND generated-index page titles
- Survives minor Docusaurus upgrades
- ~40 lines of code in a single file
- Can be added as a plugin in `docusaurus.config.ts` without creating a separate npm package

## 6. Implementation Steps

### Step 1: Create the sidebar items generator wrapper

**File:** `comapeo-docs/src/components/sidebar-i18n-fix.ts`

```ts
import fs from 'fs-extra';
import path from 'path';
import type { SidebarItem } from '@docusaurus/plugin-content-docs';

/**
 * After the default sidebar generator builds the sidebar, apply translated
 * labels from locale-specific _category_.json files.
 * 
 * Docusaurus reads _category_.json only from contentPath (English) and
 * ignores contentPathLocalized (translated). This wrapper fixes that.
 */
export async function applyLocalizedCategoryLabels<T extends SidebarItem>(
  items: T[],
  params: {
    version: { contentPath: string; contentPathLocalized?: string };
  },
): Promise<T[]> {
  const { version } = params;
  if (!version.contentPathLocalized) return items; // EN locale, nothing to patch

  async function walkItems(items: T[]): Promise<T[]> {
    return Promise.all(
      items.map(async (item): Promise<T> => {
        if (item.type !== 'category') return item;
        
        // item.source holds the folder name relative to the content path
        if ('source' in item && item.source) {
          const localizedMetaPath = path.join(
            version.contentPathLocalized!,
            item.source as string,
            '_category_.json',
          );
          if (await fs.pathExists(localizedMetaPath)) {
            const localizedMeta = await fs.readJSON(localizedMetaPath);
            const patch: Record<string, unknown> = {};
            if (localizedMeta.label) patch.label = localizedMeta.label;
            if (localizedMeta.link?.title && item.link) {
              patch.link = { ...item.link, title: localizedMeta.link.title };
            }
            if (localizedMeta.customProps?.title) {
              patch.customProps = {
                ...(item.customProps || {}),
                title: localizedMeta.customProps.title,
              };
            }
            Object.assign(item, patch);
          }
        }

        // Recurse into children
        if ('items' in item && item.items) {
          item.items = await walkItems(item.items as T[]);
        }
        return item;
      }),
    );
  }

  return walkItems(items);
}
```

### Step 2: Wire into Docusaurus config

**File:** `comapeo-docs/docusaurus.config.ts`

In the docs preset configuration, add `sidebarItemsGenerator`:

```diff
 docs: {
   path: "docs",
   sidebarPath: "./src/components/sidebars.ts",
   remarkPlugins: [remarkFixImagePaths],
   editUrl: "...",
+  sidebarItemsGenerator: async function ({item, defaultSidebarItemsGenerator, ...params}) {
+    const sidebarItems = await defaultSidebarItemsGenerator({item, ...params});
+    const { applyLocalizedCategoryLabels } = await import('./src/components/sidebar-i18n-fix');
+    return applyLocalizedCategoryLabels(sidebarItems, { version: params.version });
+  },
   lastVersion: "current",
   // ...
 },
```

### Step 3: Test

1. `docusaurus start --locale pt` -- verify sidebar shows "Visao Geral" not "Overview"
2. Navigate to a category index page -- verify `<h1>` shows the translated title
3. `docusaurus start --locale es` -- verify "Vista General" etc.
4. `docusaurus build` -- verify production build has correct labels
5. Add a new page to a section, re-run pipeline + `docs:pull`, rebuild Docusaurus -- verify sidebar updates correctly

### Step 4: Handle edge cases

- **Missing `_category_.json`:** If a section doesn't have a translated `_category_.json`, the English `_category_.json` label is used (default behavior preserved). Our pipeline already generates translated `_category_.json` for all sections in all locales.
- **Missing `contentPathLocalized`:** For EN locale, `contentPathLocalized` is undefined. The guard at the top of `applyLocalizedCategoryLabels` returns early.
- **Flattened items:** If a category is flattened (all items are drafts, or the category is empty), no label override is needed.

## 7. Fallback: If the `sidebarItemsGenerator` approach doesn't fully work

If `sidebarItemsGenerator` can't fix the `link.title` for generated-index pages (because the link object might already be resolved with permalink by the time the generator runs), use **Option A variant** -- a small Docusaurus plugin that hooks into `translateContent`:

```ts
// docusaurus-plugin-fix-category-i18n.ts
export default function () {
  return {
    name: 'fix-category-i18n',
    translateContent({ content, translationFiles }) {
      // After the docs plugin translates content, walk the loadedVersions
      // and apply localized _category_.json labels
      // This is more invasive but reaches the final sidebar + generated-index
    },
  };
}
```

The `getPathsToWatch` issue (line 141 in `index.js`) also needs to be fixed to watch `**/_category_.{json,yml,yaml}` in localized paths for hot reload. This can be addressed in the same plugin by extending `getPathsToWatch`.
