## Expected behavior — ordering

From `PRD-sidebar-ordering.md` + `TASKS.md` item 7:

- Section order is driven by Notion's EN Toggle **`Order`** property. Sections sort by that number, then positions are assigned sequentially `1..N` in `_category_.json` `position`.
- Pages within a section order by frontmatter **`sidebar_position`**, derived from the Notion `Order` property. Pages with no explicit `Order` get sequential fallback positions *after* pages that have one (`assignFallbackPositions`).
- Non-numbered "toggle-less" sections (`Overview`, `Sharing and Exporting…`) sit at the **end** (order 999).
- Pages with `section: null` (no `Content Section` in Notion) must NOT land as top-level sidebar links — they must be funneled into a fallback **`Uncategorized`** section at the highest position (999/last).
- The generated `_category_.json` must be the single source of position truth and must actually land in `comapeo-docs` (not just a temp dir).
- Acceptance order per PRD §7: Gathering(1) → Managing Data Privacy(2) → Reviewing(3) → Exchanging(4) → Miscellaneous(5) → Preparing(6) → Managing Projects(7) → Sharing(8) → Overview(9) → Uncategorized(10).

## Pipeline output & sync mechanism

- **`comapeo-content-pipeline/output/`** holds the `sync:full` dump: **295 raw page-id-named `.md`** (e.g. `1d81b081-62d5-8164-9b7e-fdc21a0e3c9b.md`), plus `manifest.json` and an `assets/` dir. Each raw `.md` carries full frontmatter (`locale`, `section`, `sidebar_position`, `status`, `content_hash`, etc.) and the real converted Markdown body.
- The pipeline does **not** persist structured `docs/{locale}/docs/{section}/{slug}.md` to disk. That layout (CLAUDE.md `R2_PATHS`: `docs/{locale}/docs/{section}/{slug}.md`) is produced transiently by the `docs:pull` CLI subcommand into a temp dir, then copied into `comapeo-docs`. So structured content only lives in `comapeo-docs`, not in the pipeline repo.
- **Sync script:** `scripts/sync-to-comapeo-docs.sh`. It:
  1. Runs `bun src/cli/index.ts docs:pull --input ./output/manifest.json --input-dir ./output --out "$TEMP_OUT" --all --clean-orphans` (temp out, default `/tmp/comapeo-docs-sync`).
  2. `rsync -av --delete` `docs/` and `i18n/` into `$DOCS_DIR` (default `../comapeo-docs`).
  3. `rm -rf .docusaurus build && npx docusaurus build`.
  4. Prints `python3 -m http.server 8765 --directory build` to preview.
  - This is the "Solution A" integration script from `PRD-sidebar-ordering.md` Phase 2 — it exists and implements the canonical `docs:pull --all --clean-orphans` → overwrite flow.

## How to run the docs site

- Runtime: **bun** (`bun.lock`, `bunfig.toml`; `fix:frontmatter` and notion scripts use `bun`). `npm`/`npx` also work for the Docusaurus commands.
- Start dev server:
  ```
  cd comapeo-docs && bun run dev          # = docusaurus start
  ```
  Locales: `bun run dev:es` / `bun run dev:pt` (`docusaurus start --locale es|pt`).
- Default URL/port: **http://localhost:3000** (Docusaurus default). `npm run dev` is equivalent.
- Build: `bun run build` (= `fix:frontmatter` → `generate:robots` → `docusaurus build`). Note `build` will currently fail/skip EN content because `docs/` is absent — run the sync script first to populate it.
